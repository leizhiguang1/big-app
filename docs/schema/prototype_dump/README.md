# v1 prototype dump — REFERENCE ONLY

Frozen snapshot of the v1 prototype Supabase database, for the new-repo agents to reference real-shape data when building v2. **Not** a migration, **not** a seed, **not** canonical.

Canonical v2 schema: [`../initial_schema.sql`](../initial_schema.sql). Canonical v2 seed: [`../seed.sql`](../seed.sql). If the dump contradicts those, canonical wins.

## What's here

| Path | What it is |
|------|------------|
| `schema_history.sql` | Concatenation of the v1 prototype's `supabase/migrations/*.sql` in chronological order. Read top-to-bottom to see how the schema evolved. |
| `data/*.json` | Full row dump of every table the anon role can read — 28 tables. Generated by `dump-anon.mjs`. **Contains real customer PII; keep repo private.** |
| `tables_inventory.md` | Auto-generated: every discovered table with status (`ok` / `missing` / `blocked`) and row count. |
| `dump-anon.mjs` | Node script that produces `data/` and `tables_inventory.md` using only the public anon key. Re-run to refresh. |

## Refreshing the dump

```bash
node docs/schema/prototype_dump/dump-anon.mjs
```

Needs Node 20+. Reads `VITE_SUPABASE_URL` / `VITE_SUPABASE_ANON_KEY` from `.env.local`. No DB password, no `pg_dump`, no `service_role` key.

**How it discovers tables** (since anon can't hit PostgREST's OpenAPI root):
1. `CREATE TABLE` + `ALTER TABLE` statements in `schema_history.sql`.
2. `.from('<table>')` calls in `src/**/*.{js,jsx,ts,tsx}` — catches tables created via Studio UI without a migration.

The union is probed via `HEAD /rest/v1/<table>` with `Prefer: count=exact`, then paginated in 1000-row windows via `Range:` headers.

**Limits**: no column types / PKs / FKs / RLS policies / functions / views / triggers — PostgREST exposes rows, not catalog metadata. For types, read `schema_history.sql`. Tables that anon can't `SELECT` show up as `blocked` with empty `data/` files.

## Key v1 → v2 discrepancies

Findings from probing the prototype that will shape the new repo:

1. **Billing storage shape — JSONB vs normalized.** [SCHEMA.md §9](../../SCHEMA.md) says v2's `billing_entries` uses a JSONB `items` array and "matches the working prototype exactly". It does not. The prototype's `appointment_billing_items` is fully normalized, one row per line. The JSONB approach is a *new* v2 design choice. Decide before building Sales whether to keep the JSONB bundle ([04-sales.md](../../modules/04-sales.md)) or mirror the normalized prototype.

2. **Sales/Payments flow doesn't exist in the prototype.** `sales` is a flat sheet. No `sale_items`, `payments`, or `cancellations` tables. The "Collect Payment" transactional service is a **net-new v2 build** — the single most important Phase 1 service and it has no working reference.

3. **Service ID type**: prototype uses text PKs (`svc-1`); v2 uses UUID.

4. **Customer ID field**: prototype has a single `identification_no`; v2 splits into `id_type` + `id_number`.

5. **Wallet balances on `customers` row**: prototype stores `cash_wallet_balance`, `beauti_points_balance`, `credit_voucher_balance`, `loyalty_points_balance` directly. v2 computes from ledger rows.

6. **Denormalized text columns**: `customer_name`, `outlet_name`, `dentist`, `service_name`, `membership_no` scattered across `appointments` and `sales`. v2 removes these in favor of JOINs.

7. **`brand_id` already in prototype**: several tables carry `brand_id: "brand-big-dental"`. v2 defers multi-tenant and drops the column.

8. **`employees.shifts` JSONB vs `employee_shifts` table**: prototype has both. The JSONB field is dead code — v2 uses only `employee_shifts` (date-based, no recurrence).

9. **Membership prefix**: prototype uses `KMD` (`KMD000019`). v2 will be brand-agnostic.

10. **`case_notes.content` vs `.body`**: prototype field is `content`. Pick one for v2 and use it consistently.

11. **Ledger-style storage for vouchers exists** (`credit_vouchers`, 9 rows) even though `payments` / `sale_items` don't. Pattern to learn from when building the payments ledger.

12. **Inventory module is real in v1** — `inventory`, `inventory_brands`, `inventory_categories`, `inventory_suppliers`, `inventory_uom`, `bundles`, `mix_match`, `promotions`, `open_vouchers`, `targeted_vouchers` all have rows. v2 plan currently treats Inventory as out-of-scope; reconcile before cutting anything.

13. **`patients` table doesn't exist** despite the recent UI rename — data is still in `customers`. No data migration happened yet.

## What the agent should do

- **Read field shapes from `data/*.json`** when implementing a service ("what does a real appointment row look like?").
- **Read `schema_history.sql`** top-to-bottom to understand the evolution of the v1 schema.
- **Do not** apply any of this as a migration or seed.
- **Do not** treat the dump as canonical — `../initial_schema.sql` is the truth.
- **Flag new contradictions** with the v2 plan in the relevant `docs/modules/*.md` file, not in this README.
