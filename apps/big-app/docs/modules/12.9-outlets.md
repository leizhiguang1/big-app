# Module: Outlets

> Status: Built — minimal v1 scope (outlets + rooms CRUD).
> Migration: `0013_outlets`. Live route: `/config/outlets` (Config → redirects here).

## Overview

Outlets are the clinic branches. They are a **lightweight foundation entity** — v1 only needs outlets to do three things:

1. **Scope appointments** — every appointment belongs to one outlet; the calendar is always filtered by outlet.
2. **Assign staff** — employees are linked to one or more outlets so the appointment screen can pick the right staff for the right branch.
3. **Mark the customer's home branch** — customers remember which outlet they first registered at (useful for reporting and membership numbering).

We deliberately keep the outlet record small. KumoDent's Outlets screen has extras (daily summary email, print type, security, contact person, floors, district) — none of those are needed for Phase 1 and they can be added later without schema churn.

## Identifiers (id vs code)

Two different identifiers, two different jobs:

| | `outlets.id` | `outlets.code` |
|---|---|---|
| Type | `uuid` | `text` |
| Example | `f1000000-0000-0000-0000-000000000001` | `BDK` |
| Where it lives | DB primary key + foreign keys | URLs (`/o/BDK/...`), invoice numbering, customer code prefix |
| User-visible | No | Yes |
| Mutable | No | No (frozen at create — historical codes refer back) |

**Rule of thumb:** when you need to reference an outlet, use `id` everywhere except URLs and human-readable identifiers. `code` exists purely so URLs and customer membership numbers don't expose UUIDs.

### Uniqueness (migration `0099_outlets_code_unique_per_brand`)

```
unique (brand_id, code)              ← per-brand, not global
check  (code ~ '^[A-Z0-9]{2,6}$')    ← URL-safe, no dashes
```

That means:
- **Big Dental can have outlet `MAIN` and Test Dental can also have `MAIN`** — they don't collide.
- Codes must be 2–6 chars, uppercase letters or digits only. Dashes are forbidden because the URL parser splits on `/`, and a code like `BD-K` would break `/o/BD-K/customers` if anyone ever tried lowercase routing or a different segment shape.
- The form's Zod schema (`lib/schemas/outlets.ts`) enforces the same regex front-end so users get a friendly error before the DB does.

### Why we don't shorten the UUID

The UUID is invisible in this app — URLs use `code`, the UI never shows a UUID, and the only places it appears are FK columns and DB inspectors. Switching to nanoid / ULID / short prefixes would mean rewriting every FK in the schema for zero user-facing benefit. If a short visible identifier is ever needed (e.g. a public share link), add a separate `public_slug` column to that one entity rather than changing the PK.

## Workflows & Status Transitions

Outlets don't have a status machine — just `is_active = true | false`.

```
active ⇄ inactive
```

Deactivating an outlet:
- Hides it from appointment creation, customer registration, and staff assignment pickers
- Does NOT hide or delete existing appointments, customers, or sales tied to it (historical records must remain readable)

## Business Rules

- Outlet `code` is set at creation and never changed (membership numbers depend on it).
- Outlets can never be hard-deleted — only deactivated. Schema enforces this via `ON DELETE RESTRICT` on every FK that references `outlets`.
- A customer has exactly one `home_outlet_id`, set at registration. They can still visit and book at any outlet.
- An employee can be assigned to 1..N outlets via `employee_outlets`.
- An appointment belongs to exactly one outlet.
- A room belongs to exactly one outlet. Rooms are optional on appointments (some services don't need a room).

## Gaps & Improvements Over KumoDent

- **Dropped all the noise:** no contact-person field, no floors, no district, no SKU Code / Outlet ID split — just `code` and UUID.
- **Rooms are a proper table** instead of `text[]` on the outlet. Lets us FK rooms from appointments and add per-room attributes later.
- **No brand column yet.** KumoDent has no brand concept; we'll introduce one when multi-tenant lands.
- **Deferred tabs** — Daily Summary Email, Print Type, Security — all Phase 2+. Notes for when we revisit:
  - Daily Summary Email → probably belongs in a Notifications/Config module
  - Print Type → receipt template setting, belongs in Sales/Config
  - Security → covered by Auth + role_permissions

## Schema Notes

Live shape (after migration `0099_outlets_code_unique_per_brand`):

```sql
CREATE TABLE outlets (
  id          UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  brand_id    UUID NOT NULL REFERENCES brands(id) ON DELETE RESTRICT,
  code        TEXT NOT NULL,
  name        TEXT NOT NULL,
  -- … address, phone, email, invoice/profile fields …
  is_active   BOOLEAN NOT NULL DEFAULT true,
  created_at  TIMESTAMPTZ NOT NULL DEFAULT now(),
  updated_at  TIMESTAMPTZ NOT NULL DEFAULT now(),

  CONSTRAINT outlets_brand_code_unique UNIQUE (brand_id, code),
  CONSTRAINT outlets_code_format CHECK (code ~ '^[A-Z0-9]{2,6}$')
);
```

Migration history (latest first):
- `0099_outlets_code_unique_per_brand` — drop global `unique(code)`, add `unique(brand_id, code)` + `check(code ~ '^[A-Z0-9]{2,6}$')`
- `0074_core_brand_id` — add `brand_id` (multi-brand rollout)
- `0066_outlets_code_immutable` — trigger to block UPDATEs of `code`
- `0070_outlets_wa_connection_id` — partial unique on `wa_connection_id`
- `0028_outlets_default_room` — auto-create a default room on insert
- `0013_outlets` — initial table
