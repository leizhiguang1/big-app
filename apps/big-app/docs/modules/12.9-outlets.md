# Module: Outlets

> Status: Built — minimal v1 scope (outlets + rooms CRUD).
> Migration: `0013_outlets`. Live route: `/config/outlets` (Config → redirects here).

## Overview

Outlets are the clinic branches. They are a **lightweight foundation entity** — v1 only needs outlets to do three things:

1. **Scope appointments** — every appointment belongs to one outlet; the calendar is always filtered by outlet.
2. **Assign staff** — employees are linked to one or more outlets so the appointment screen can pick the right staff for the right branch.
3. **Mark the customer's home branch** — customers remember which outlet they first registered at (useful for reporting and membership numbering).

We deliberately keep the outlet record small. KumoDent's Outlets screen has extras (daily summary email, print type, security, contact person, floors, district) — none of those are needed for Phase 1 and they can be added later without schema churn.

**Form fields (v1):**

## Workflows & Status Transitions

Outlets don't have a status machine — just `is_active = true | false`.

```
active ⇄ inactive
```

Deactivating an outlet:
- Hides it from appointment creation, customer registration, and staff assignment pickers
- Does NOT hide or delete existing appointments, customers, or sales tied to it (historical records must remain readable)

## Business Rules

- Outlet `code` is set at creation and never changed (membership numbers depend on it).
- Outlets can never be hard-deleted — only deactivated. Schema enforces this via `ON DELETE RESTRICT` on every FK that references `outlets`.
- A customer has exactly one `home_outlet_id`, set at registration. They can still visit and book at any outlet.
- An employee can be assigned to 1..N outlets via `employee_outlets`.
- An appointment belongs to exactly one outlet.
- A room belongs to exactly one outlet. Rooms are optional on appointments (some services don't need a room).

## Relationships to Other Modules

| Related Module | Relationship | Details |
|---------------|-------------|---------|
| Employees | outlet ↔ employees | Many-to-many via `employee_outlets` |
| Customers | outlet ← customer | `customers.home_outlet_id` (RESTRICT) |
| Appointments | outlet ← appointment | `appointments.outlet_id` (RESTRICT) |
| Rooms | outlet → rooms | One-to-many, cascade delete |
| Sales | outlet ← sales_orders | `sales_orders.outlet_id` (RESTRICT) |
| Roster | outlet ← shifts | `employee_shifts.outlet_id` — a shift is always at a specific outlet |

## Gaps & Improvements Over KumoDent

- **Dropped all the noise:** no contact-person field, no floors, no district, no SKU Code / Outlet ID split — just `code` and UUID.
- **Rooms are a proper table** instead of `text[]` on the outlet. Lets us FK rooms from appointments and add per-room attributes later.
- **No brand column yet.** KumoDent has no brand concept; we'll introduce one when multi-tenant lands.
- **Deferred tabs** — Daily Summary Email, Print Type, Security — all Phase 2+. Notes for when we revisit:
  - Daily Summary Email → probably belongs in a Notifications/Config module
  - Print Type → receipt template setting, belongs in Sales/Config
  - Security → covered by Auth + role_permissions

## Schema Notes

Already in [schema/initial_schema.sql](../schema/initial_schema.sql) — see the `outlets` and `rooms` tables. Only change from the initial draft: **dropped `brand_id`** (premature).

```sql
-- Simplified outlets table (final)
CREATE TABLE outlets (
  id          UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  code        TEXT NOT NULL UNIQUE,   -- used as membership_no prefix
  name        TEXT NOT NULL,

  address1    TEXT,
  address2    TEXT,
  city        TEXT,
  state       TEXT,
  postcode    TEXT,
  country     TEXT DEFAULT 'Malaysia',

  phone       TEXT,
  email       TEXT,

  is_active   BOOLEAN NOT NULL DEFAULT true,
  created_at  TIMESTAMPTZ NOT NULL DEFAULT now(),
  updated_at  TIMESTAMPTZ NOT NULL DEFAULT now()
);
```

No triggers, no generated columns, no sequences. Outlets are the simplest table in the system.
