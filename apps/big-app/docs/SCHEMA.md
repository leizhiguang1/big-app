## Conventions every new table follows

1. `id uuid primary key default gen_random_uuid()`
2. `code text unique not null` **only on tables where users see, search,
   or quote the identifier** (customers, sales orders, employees on
   payslips, etc.). Internal lookup tables ‚ÄĒ roles, positions, categories,
   junction tables, settings ‚ÄĒ do not get a code column. When a table does
   have one, format and sequence width are decided per-entity (`EMP-0001`,
   `CUS-00000001`), generated by a `BEFORE INSERT` trigger from a per-table
   sequence ‚ÄĒ never set in app code. Default to skipping; codes are easy
   to add later, hard to remove.
3. `created_at timestamptz not null default now()`
4. `updated_at timestamptz not null default now()` + shared `set_updated_at`
   `BEFORE UPDATE` trigger
5. `is_active bool default true` **only** when a concrete use case forces
   soft-delete (FK breakage, audit history). Default is hard delete with
   `ON DELETE RESTRICT`. Bias toward skipping ‚ÄĒ easy to add later. Existing
   tables that already carry it keep it; this rule applies forward only.
6. RLS enabled the moment the table is created. Pre-auth: permissive
   `anon` policy marked `-- TEMP: pre-auth, replace when login lands`.
   Post-auth: tightened per module.
7. No denormalized text columns ‚ÄĒ always JOIN.
8. **Brand scoping (added 2026-04-21):** top-level business entities
   carry `brand_id uuid not null references brands(id) on delete restrict`.
   Children/junctions inherit via parent FK ‚ÄĒ do NOT denormalize.
   Lookups stay global. Full rules, tier classification, migration
   template, service patterns, and common mistakes:
   [BRAND_SCOPING.md](./BRAND_SCOPING.md) ‚ÄĒ **read before adding a new
   table**.

The shared `set_updated_at()` function and `gen_code(prefix, seq_name)`
helper are introduced in the first MCP migration before any feature table.

## Schemas owned by other services

Big-app is not the only writer on its Supabase project. **whatsapp-crm** (a separate Node+Baileys process, fresh clone of the reference whatsapp-crm repo, deployed to its own Railway service) shares this project and owns:

- **Every table in the `wa_crm` schema** ‚ÄĒ connections, message cache, webhook log, automation templates, automation runs, scheduled sends, chat-CRM tables (WA labels, unknown senders, conversation tags authored from chat). Defined and migrated from the whatsapp-crm repo, not here.
- **A Supabase Storage bucket for chat media** ‚ÄĒ inbound media uploads from WhatsApp. Created and managed by whatsapp-crm.

A predecessor service called **wa-connector** (pure transport, all `public.wa_*` tables) is being decommissioned in favor of whatsapp-crm. For as long as `public.wa_*` tables still exist (`wa_api_keys`, `wa_connections`, `wa_webhook_log`, `wa_message_log`, `wa_chat_cache`, `wa_message_cache` plus the `wa-media` Storage bucket), the same ownership rules apply ‚ÄĒ they are out of big-app's scope. They disappear when cutover is complete.

**Rules (load-bearing ‚ÄĒ these are how two services sharing one DB don't step on each other):**

1. Do **not** include external-service tables (`wa_crm.*` or legacy `public.wa_*`) in big-app's table inventory below.
2. Do **not** create migrations in this repo that `CREATE`, `ALTER`, `DROP`, or otherwise touch them ‚ÄĒ not even to add an RLS policy or a comment. Any schema change is made in the whatsapp-crm repo and applied from there.
3. Do **not** read from them in `lib/services/**` code. Big-app's services only touch tables it owns. Big-app's WhatsApp / Conversations modules talk to whatsapp-crm over its HTTP API and mirror what they need into their own `public.*` tables.
4. Their RLS state and naming are intentionally out of scope for this repo. Don't "fix" them here.

**No naming collision.** big-app's Conversations module ([modules/11-conversations.md](./modules/11-conversations.md)) uses channel-agnostic names in `public` (`conversations`, `conversation_messages`, `channel_accounts`, `conversation_channels`) ‚ÄĒ intentionally no `wa_` prefix, so there's zero ambiguity in `list_tables` output. External-service tables are either in a separate schema (`wa_crm.*`) or use the `wa_` prefix in `public` during the transition.

See [ARCHITECTURE.md ¬ß2 + ¬ß2.1](./ARCHITECTURE.md) and [docs/WA_CRM_INTEGRATION.md](./WA_CRM_INTEGRATION.md) for the full decision and the communication rules.

## Storage (Supabase Storage buckets)

Big-app uses Supabase Storage for binaries. Two buckets, chosen by read-access pattern:

- **`media`** ‚ÄĒ public read, 5 MB, image MIME only. Employee/customer avatars, service/product/outlet photos. Public URL derived via `lib/storage/urls.ts ‚Üí mediaPublicUrl(path)`.
- **`documents`** ‚ÄĒ private, 20 MB, images + PDFs. ID scans, clinical photos, signed forms, receipts. Read via short-lived signed URLs (`lib/services/storage.ts ‚Üí createSignedReadUrl`).

Domain tables carry the **path**, not the URL ‚ÄĒ e.g. `employees.profile_image_path`, `customers.profile_image_path`. Path convention: `<entity>/<entity_id>/<yyyymmdd>-<uuid>.<ext>`.

Upload flow is direct-to-Supabase (browser ‚Üí Supabase, never through Next) via signed upload URLs minted by `lib/actions/storage.ts ‚Üí requestMediaUploadUrlAction`. The reusable UI surface is `components/ui/image-upload.tsx`.

RLS on `storage.objects` currently uses temp permissive policies (both `anon` and `authenticated`) per bucket, marked `-- TEMP: pre-auth tightening` ‚ÄĒ same pattern as every other table, tightened when the real auth model lands.

Full rationale in [ARCHITECTURE.md ¬ß11](./ARCHITECTURE.md#11-file-storage--supabase-storage-two-buckets-path-in-db).

## Key Design Decisions

### 1. UUID PK + separate human-readable code column

Every table uses `id uuid primary key default gen_random_uuid()` *plus* a
separate `code text unique not null` column for the human-readable
identifier (`EMP-000001`, `CUS-000001`, etc.). The code is generated by a
trigger from a per-table sequence ‚ÄĒ never set in app code, never used as a
foreign key. Internal joins use `id`; UI / receipts / search use `code`.

This fixes the v1 prototype's mess of mixed `uuid` / `text` / `integer` PKs
*and* gives end users a stable, friendly identifier without coupling it to
the database PK. Renaming the format later (`EMP-000001` ‚Üí `EMP-2026-001`)
changes one trigger and one sequence ‚ÄĒ not a primary key.

### 2. No denormalized text fields

Removed `customer_name`, `employee_name`, `membership_no`, `outlet_name`, etc. from all tables that stored them alongside FK IDs. The v2 schema uses JOINs instead.

**Impact on frontend:** Queries that previously returned flat rows with names now need JOINs. Use Supabase's `.select('*, customer:customers(first_name, last_name)')` syntax and compose the display name at the render layer.

### 3. Rooms as a proper table

Replaced `rooms TEXT[]` on outlets with a `rooms` table. Benefits:
- `room_id` FK on appointments (was `chair TEXT`)
- Can track room status, capacity, equipment later
- Cross-industry: label as "room", "chair", "bed", "station" in the UI

### 3b. Outlets kept deliberately minimal

The `outlets` table only carries what Phase 1 actually uses: `code`, `name`,
address, phone, email, `is_active`. Dropped from earlier drafts: `brand_id`
(no brand concept yet), contact-person, floors, district, per-outlet hours.
KumoDent's extras (Daily Summary Email, Print Type, Security tabs) are deferred.
See [modules/01-outlets.md](./modules/01-outlets.md).

### 4. Service categories as a separate table

Replaced `category TEXT` on services with a `service_categories` table. Supports CRUD in the services UI and consistent categorization.

### 5. Generated columns

Generated (computed) columns that stay in sync automatically:
- `appointment_line_items.total` ‚ÄĒ `qty * unit_price` (per line; normalized row-per-line, see ¬ß9).
- `sale_items.total` ‚ÄĒ `quantity * unit_price - discount`
- `sales_orders.outstanding` ‚ÄĒ `total - amount_paid`

**Not a generated column:** `customers.display_name` was proposed but never implemented ‚ÄĒ the app concatenates `first_name + last_name` at the query / component layer instead. If we later need indexable full-name search we'll add it via a stored generated column or a functional index; until then, no column.

### 6. Wallet balances removed from customers

Removed `cash_wallet_balance`, `beauti_points_balance`, `credit_voucher_balance`, `loyalty_points_balance` from customers table. These will be computed from transaction tables when loyalty features are built in Phase 2.

**Resolved 2026-04-24:** The cash wallet shipped in Phase 1 as a separate `customer_wallets` + `wallet_transactions` pair (rows 19 and 20 above) ‚ÄĒ `balance` is a stored running total plus an append-only signed-amount ledger, not a denormalized column on `customers`. The other three balances (beauti points, credit vouchers, loyalty points) remain Phase-2 concerns. Later that day (**v2 pivot**) Cash Wallet became a pinned `inventory_items` row (sku `CASH_WALLET`) sold exclusively via a single-line SO, with FIFO tranche tracking via `wallet_transactions.amount_remaining` + new `wallet_allocations` table (row 20a). See [modules/15-cash-wallet.md](./modules/15-cash-wallet.md).

### 7. Customer & employee ID type

Both `customers` and `employees` use the same `(id_type, id_number)` pair, where `id_type` is `'ic'` or `'passport'`. Replaces the ambiguous single `identification_no` field from v1. Phase 1 only validates Malaysian IC format (`YYMMDD-PB-###G`); foreign nationals use `id_type = 'passport'` with free-form numbers. A partial unique index on `(id_number) where id_type = 'ic'` is applied to both tables to block duplicate ICs; passport numbers are intentionally not uniqued because they can collide across issuing countries. The employees column was originally named `identification_no` and was renamed in the `unify_id_number_columns` migration to match customers.

### 8. Date-based shifts (no recurrence)

`employee_shifts` stores one row per employee per date. No `day_of_week` or `repeat_type` columns. The app handles "copy week" / "repeat schedule" at the application level. Simpler and more explicit.

### 9. Appointment_line_items ‚Üí sale_items, both normalized

**Naming note (2026-04-15):** The table was originally called `billing_entries`. It was renamed to `appointment_line_items` to reflect its true dual role: it is *both* the clinical record of what was performed on an appointment AND the billing cart used at payment time. The old name implied "billing only" which misled readers into expecting a separate "services performed" table. The rename is data-layer only ‚ÄĒ UI components (`BillingTab`, `BillingSection`) and the user-facing "Billing" tab label kept their names because that's the word staff uses.

**Note:** An earlier draft of this document proposed a two-tier design where the table wrapped a JSONB `items` array (matching the prototype's `useBillingEntries` hook) and `sale_items` was the normalized destination. That asymmetry was dropped during implementation. Both tables are now normalized row-per-line ‚ÄĒ the JSONB story is historical.

Current shape:

- **`appointment_line_items`** ‚ÄĒ one row per line item on an appointment. Columns: `appointment_id`, `item_type` (`service` | `product` | `charge`), `service_id` (set only when `item_type='service'`), `product_id` (set only when `item_type='product'`, FK ‚Üí `inventory_items`), `description`, `qty`, `unit_price`, `total`, plus standard audit columns. A CHECK (`appointment_line_items_type_ref_check`) enforces the FK/type pairing so rows can't drift. Staff add/edit lines freely during the visit; there is no "Save Billing" batching layer. The v1 Billing tab exposes the Services and Products tabs of the picker ‚ÄĒ Laboratory / Vaccinations / Other Charges are placeholder tabs. Has one child table (CASCADE on delete):
  - `appointment_line_item_incentives` ‚ÄĒ per-line employee attribution (`employee_id`). Multiple rows allowed; `UNIQUE (line_item_id, employee_id)`. No commission calculation in v1 ‚ÄĒ this is the data plumbing for a future Commission module.
- **Consumables** are NOT a child table on the appointment side. They're a catalog-level property stored in the `service_inventory_items` junction (service ‚Üí inventory item + `default_quantity`); the appointment-side `ConsumablesCard` reads that junction per service line and shows each linked item with its computed deduction (`default_quantity √ó billed_qty`). Deduction against `inventory_items.stock` happens on Collect Payment via the RPC. An earlier revision of this refactor created an `appointment_line_item_consumables` table with a per-line editor; it was dropped after clarifying that consumables are a service property, not per-visit. The former free-text `services.consumables` column was also dropped (2026-04-17) once the junction shipped.
- **`sale_items`** ‚ÄĒ normalized rows on a committed sales order. Created when staff clicks "Collect Payment" ‚ÄĒ line items are snapshotted into sale items and the sales order is committed in the same transaction. Incentive rows are **not** copied ‚ÄĒ they remain attached to the originating line item as a historical record.

Why keep both tables even though they look similar: `appointment_line_items` are mutable workspace state tied to an open appointment (add, delete, edit freely); `sale_items` are the immutable financial record once the SO is committed. Editing a committed SO goes through a cancellation/reissue flow, not an in-place edit of `sale_items`.

**Service-only children invariant.** Incentives must attach to a line item with `item_type = 'service'`. Postgres CHECK constraints can't express this (can't subquery the parent row), so it's enforced in `lib/services/appointment-line-items.ts` via `assertServiceLineItem()`. A trigger is easy to add later if belt-and-braces is wanted.

**Price override:** dentist can type any `unit_price` on a line ‚ÄĒ no constraint against the service catalog. Same freedom carries through to `sale_items`.

### 10. Cancellations ‚ÄĒ deferred

An earlier draft of this doc planned a dedicated `cancellations` table with CN-numbered records (matching KumoDent's CN document generation pattern). **Not built in Phase 1.** Cancellation today is a status transition on `sales_orders` ‚ÄĒ `status` supports `cancelled` and `void` alongside `draft` / `completed`. The separate audit table + CN document generation is revisited if/when the finance team asks for it.

### 11. Appointment status machine

```
pending ‚Üí confirmed ‚Üí arrived ‚Üí started ‚Üí billing ‚Üí completed
                                                  ‚Üí noshow
```

Live enum values: `pending`, `confirmed`, `arrived`, `started`, `billing`, `completed`, `noshow`. Default is `pending`. **There is no `cancelled` status on appointments** ‚ÄĒ it was removed in migration `appointments_drop_cancelled_status`; deleting an appointment is a hard delete. CHECK constraint enforces valid values.

(An earlier draft of this doc described a simpler `scheduled ‚Üí confirmed ‚Üí in_progress ‚Üí completed / cancelled / no_show` machine. That was the v1 target; the live shape grew during the build to match the operational stages the front desk actually uses.)

### 12. RLS ‚ÄĒ temporary permissive policies during Phase 1

Every table has RLS enabled from the moment it's created. Until per-role tightening lands module-by-module, each table carries the temp policy pair mandated by CLAUDE.md rule 6:

```sql
create policy "<table> anon all" on public.<table>
  for all to anon using (true) with check (true);
create policy "<table> authn all" on public.<table>
  for all to authenticated using (true) with check (true);
```

Both policies are marked `-- TEMP: pre-auth tightening`. When a table is tightened per role, the temp pair is dropped and replaced in the same migration. Rationale for the dual-role pair (anon + authenticated) is in CLAUDE.md rule 6 ‚ÄĒ single-role policies silently return empty rows to the wrong session. Per-role policies (receptionist can only view certain data) land during the Auth module deep-dive.

## FK Delete Behavior

| Pattern | Behavior | Rationale |
|---------|----------|-----------|
| Dependent records ‚Üí parent | `CASCADE` | If parent is deleted, children go too (billing items ‚Üí appointment, sale items ‚Üí SO, payments ‚Üí SO) |
| Soft references ‚Üí entity | `SET NULL` | Preserve the record, null out the reference (appointment ‚Üí customer, appointment ‚Üí employee) |
| Hard dependencies ‚Üí entity | `RESTRICT` | Prevent deletion (appointment ‚Üí outlet, SO ‚Üí outlet) |

## Naming Conventions

## Seed Data

**Source of truth:** [schema/seeds/](./schema/seeds/) ‚ÄĒ one SQL file per module. Each file is applied to the cloud DB as a dedicated `000N_<module>_seed` migration via the Supabase MCP and kept byte-identical to the migration body. Seed files use deterministic UUIDs and `ON CONFLICT (id) DO NOTHING` so they're safe to re-run.

[schema/seed.sql](./schema/seed.sql) (monolithic) is **reference only** ‚ÄĒ paired with the unapplied `initial_schema.sql`, kept for historical context, never updated.
